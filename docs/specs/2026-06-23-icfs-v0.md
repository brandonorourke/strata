# Spec: ICFS v0 (first primary-source module)

Status: draft  
Owner: Brandon  
Created: 2026-06-23  
Context: `docs/decisions.md` (2026-06-16), `docs/customer_conversations/2026-06-16-carronade-stas-intro-call.md`

## Goal

Monitor FCC's ICFS (International Communications Filing System — satellite earth stations, space stations, Section 214 authorizations, submarine cable landing licenses, etc.) for a small watchlist of companies, and produce a digest of what changed, with citations back to the source filing.

This is the **confirmed first build** of the primary-source pivot (decisions.md, 2026-06-16) — a real customer wants this for his Viasat equity position, checks the portal manually today and calls it "impossible to navigate," and no visible incumbent does FCC-for-investors monitoring. UCC/credit is a separate, gated second module (not in scope here) — see `docs/decisions.md`.

## Scope (v0)

In scope:
- Ingest **all** ICFS filings — not filtered by company at the source level. This builds the actual mirror/moat asset (decisions.md: "entity resolution + cross-source assembly that compounds"), the same way `ingest_rss.py` pulls full feeds today rather than filtering at fetch time.
- The watchlist (v0: hardcoded to one name, Viasat) only governs what gets surfaced in the digest — it's a downstream filter on top of full ingestion, not an ingestion-time filter.
- Reuse the existing entity/event pipeline rather than building a parallel system.
- Produce a digest: what's new/changed for watched entities since last run, with a citation (source filing link) per item.

Explicitly out of scope for v0:
- Multi-customer / multi-tenant watchlists, auth, or any UI for customers to manage their own watchlist (no auth exists anywhere in the app yet — out of scope here too).
- UCC, mortgage/recording, or court docket ingestion (the gated second module).
- Automated alerting/email delivery of the digest — v0 just needs the digest to exist and be readable (a page or a generated markdown file is enough).
- Broader FCC RSS (dockets, rulemakings, commissioner statements) — sequenced after ICFS per `to_build.md`, treated as a separate, smaller follow-on ingestion source once ICFS works end to end.

## Architecture decision

Per `docs/decisions.md` (2026-06-16): **one engine, two adapters** — domain differences should live only in ingestion, not in extraction/storage/scoring. Concretely, that means:

- **Reuse `news_articles` + the existing pipeline shape** (`fetch_html` → `clean_text` → `llm_raw` → `extract_entities`) rather than introducing a parallel "documents" table. An ICFS filing fits the existing columns fine: `url`, `title`, `published_at`, `raw_html`/`clean_text`, `llm_raw`. Add a new `NewsSource` enum value (e.g. `FCC_ICFS`) the same way `SEC_PRESS_RELEASES` etc. were added in migration `0014` — append-only, no reordering, per the existing schema convention.
- **New ingestion adapter only**: `apps/ingest/ingest_icfs.py`, mirroring `ingest_rss.py`'s shape but fetching/parsing ICFS filings instead of an RSS feed (see Open Questions below — the exact fetch mechanism isn't confirmed yet).
- **New concept, not in the current schema: watchlist + pending entities.** Today's entity model only does global time-window clustering (`docs/decisions.md`, 2026-02-05) — there's no notion of "entities a specific customer cares about." v0 needs:
  - A `watchlist_entries` table: `id`, `label` (free text for v0, e.g. "Viasat"), `legal_name_normalized` (to match against `extracted_entities`/`canonical_entities`), `status` (`watching` | `pending`), `created_at`.
  - Resolution step after `extract_entities`/`link_entities`: when a canonical entity matches a `watching` watchlist entry, mark the filing/event as relevant to that entry. If an ICFS filing surfaces an entity not yet on the watchlist (e.g. a subsidiary or affiliate), insert it as `pending` rather than dropping it — consistent with the existing "prefer duplicates over wrong merges" / conservative-canonicalization philosophy.
- **Digest generation reuses the existing "evidence-backed brief" backlog item** (`to_build.md`: "second LLM pass per top cluster... what happened, why it matters, what to watch"), scoped to watchlist entries instead of top global clusters.

## Data model changes

New table: `watchlist_entries`
- `id` (PK)
- `label` (text, not null) — display name, e.g. "Viasat"
- `legal_name_normalized` (text, not null) — for matching against `extracted_entities`/`canonical_entities`
- `status` (text, not null, default `'watching'`) — `watching` | `pending`
- `created_at` (timestamptz, default now())

Schema convention / workflow: write the migration (`migrations/0015_add_watchlist_entries.sql`) and the corresponding `WatchlistEntry` model in `strata_core/models.py` as code. **Do not hand-generate `strata_core/schema.local.sql`** — per the existing convention (`docs/decisions.md`, 2026-02-07), that file is a literal `pg_dump` snapshot. The sequence is: (1) write the migration + model code, (2) run the migration against the dev DB, (3) regenerate the snapshot with `pg_dump strata --schema-only --no-owner > strata_core/schema.local.sql`. Steps 2–3 are run directly against the dev DB, not authored by hand.

`NewsSource` enum: add `FCC_ICFS` (and, if needed once the fetch mechanism is confirmed, a distinct value if ICFS turns out to need more than one logical source, e.g. a mirror vs. the official portal).

## Build order (from `to_build.md`)

1. Empty deployed skeleton — already done (app is live on Railway); this step is about confirming the new ingestion script + table deploy cleanly, not building UI yet.
2. Ingest everything, watch one name — `ingest_icfs.py` pulls all ICFS filings (unfiltered) into `news_articles` via the new adapter, through the existing extraction pipeline. `watchlist_entries` is seeded with just one row (Viasat) to prove the resolution/matching step against the full, real mirror.
3. Produce a digest — a simple page or generated markdown showing new/changed filings matched to the watchlist since last run, each with a citation back to the source filing and a short evidence-backed summary (what happened, why it matters, what to watch).
4. Point at the customer's actual watchlist — once the mechanics work for Viasat, expand `watchlist_entries` to whatever names the customer actually wants monitored. Ingestion itself doesn't change at this step — it was already pulling everything.

## Open questions (resolve before/during step 2)

- **ICFS access mechanism is unconfirmed.** The official portal (`fcc.gov/icfs`) requires an FRN/CORES login to *file*, but it's unclear whether searching/viewing filings by company name also requires login. A third-party mirror, `fcc.report/IBFS/`, replicates filings publicly and advertises RSS, but it's unconfirmed whether that RSS can be filtered per-company or is just an unfiltered firehose requiring client-side filtering. **First technical task: spend a short timebox confirming which of these is actually usable before writing `ingest_icfs.py` against it.**
- Filing cadence/volume for a single company (e.g. Viasat) is unknown — affects how often the ingestion adapter needs to poll.
- Whether ICFS filings come with a stable `published_at`-equivalent timestamp and a stable per-filing URL (needed for the existing `news_articles.url` unique constraint and de-dup logic) needs to be confirmed against real filings, not assumed.

## Definition of done (v0)

- `watchlist_entries` exists and is seeded with at least one entry (Viasat).
- `ingest_icfs.py` pulls real ICFS filings for that entry into `news_articles` (tagged `FCC_ICFS`), and they flow through the existing `fetch_html` → `clean_text` → `llm_raw` → `extract_entities` → `link_entities` pipeline without code changes to those steps.
- A digest (page or generated doc) lists new/changed filings for the watchlist with a citation and short summary per item.
- The customer (Stas) has seen the digest for at least one real name and given feedback.
