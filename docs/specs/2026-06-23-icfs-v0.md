# Spec: ICFS v0 (first primary-source module)

Status: draft  
Owner: Brandon  
Created: 2026-06-23  
Context: `docs/decisions.md` (2026-06-16), `docs/customer_conversations/2026-06-16-carronade-stas-intro-call.md`

## Goal

Monitor FCC's ICFS (International Communications Filing System — satellite earth stations, space stations, Section 214 authorizations, submarine cable landing licenses, etc.), and surface it as a browsable, filterable, cited list — better than the official portal, starting with the filing category where the official portal is worst (Pleadings & Comments has no company-name index).

This is the **confirmed first build** of the primary-source pivot (decisions.md, 2026-06-16) — a real customer wants this for his Viasat equity position, checks the portal manually today and calls it "impossible to navigate," and no visible incumbent does FCC-for-investors monitoring. UCC/credit is a separate, gated second module (not in scope here) — see `docs/decisions.md`.

## Scope (v0)

In scope:
- Ingest **all** ICFS filings — not filtered by company at the source level. This builds the actual mirror/moat asset (decisions.md: "entity resolution + cross-source assembly that compounds"), the same way `ingest_rss.py` pulls full feeds today rather than filtering at fetch time.
- The watchlist (v0: hardcoded to one name, Viasat) only governs what gets surfaced as "filtered" in the UI — it's a downstream filter on top of full ingestion, not an ingestion-time filter.
- Reuse the existing entity/event pipeline rather than building a parallel system.
- UI: a full, paginated list of ingested filings (same pattern as `/admin/articles`), plus a dropdown to filter the list down to just the hardcoded watchlist names (Viasat for now). This is the core v0 surface — see "ICFS filing categories" below for why this alone beats the official portal for at least one filing type.
- Citation per item back to the source filing, same evidence-backed pattern used elsewhere in the app.

Explicitly out of scope for v0:
- Multi-customer / multi-tenant watchlists, auth, or any UI for customers to manage their own watchlist (no auth exists anywhere in the app yet — out of scope here too).
- UCC, mortgage/recording, or court docket ingestion (the gated second module).
- Automated alerting/email delivery — v0 just needs the filterable list UI to exist; no push notifications.
- The evidence-backed LLM summary per filing — fast-follow, not required for v0's definition of done.
- Broader FCC RSS (dockets, rulemakings, commissioner statements) — sequenced after ICFS per `to_build.md`, treated as a separate, smaller follow-on ingestion source once ICFS works end to end.

## Architecture decision

Per `docs/decisions.md` (2026-06-16): **one engine, two adapters** — domain differences should live only in ingestion, not in extraction/storage/scoring. Concretely, that means:

- **Reuse `news_articles` + the existing pipeline shape** (`fetch_html` → `clean_text` → `llm_raw` → `extract_entities`) rather than introducing a parallel "documents" table. An ICFS filing fits the existing columns fine: `url`, `title`, `published_at`, `raw_html`/`clean_text`, `llm_raw`. Add a new `NewsSource` enum value (e.g. `FCC_ICFS`) the same way `SEC_PRESS_RELEASES` etc. were added in migration `0014` — append-only, no reordering, per the existing schema convention.
- **New ingestion adapter only**: `apps/ingest/ingest_icfs.py`, mirroring `ingest_rss.py`'s shape but fetching/parsing ICFS filings instead of an RSS feed (see Open Questions below — the exact fetch mechanism isn't confirmed yet).
- **New concept, not in the current schema: watchlist + pending entities.** Today's entity model only does global time-window clustering (`docs/decisions.md`, 2026-02-05) — there's no notion of "entities a specific customer cares about." v0 needs:
  - A `watchlist_entries` table: `id`, `label` (free text for v0, e.g. "Viasat"), `legal_name_normalized` (to match against `extracted_entities`/`canonical_entities`), `status` (`watching` | `pending`), `created_at`.
  - Resolution step after `extract_entities`/`link_entities`: when a canonical entity matches a `watching` watchlist entry, mark the filing/event as relevant to that entry. If an ICFS filing surfaces an entity not yet on the watchlist (e.g. a subsidiary or affiliate), insert it as `pending` rather than dropping it — consistent with the existing "prefer duplicates over wrong merges" / conservative-canonicalization philosophy.
- **Digest generation reuses the existing "evidence-backed brief" backlog item** (`to_build.md`: "second LLM pass per top cluster... what happened, why it matters, what to watch"), scoped to watchlist entries instead of top global clusters.

## ICFS filing categories — confirmed via live API (2026-06-23)

ICFS indexes filings into four lists, each backed by a distinct ServiceNow table, all reachable through the same generic widget endpoint (see "Access mechanism" below) by changing the request payload's `table`/`fields`:

| Category | Table | Fields (index columns) | Confirmed row count |
|---|---|---|---|
| Recent Filings | `x_fmc_ibfs_base_table` | `number, call_sign, applicant_name, submission_date` | 140,540 |
| Recent Pleadings & Comments | `x_fmc_ibfs_pleadings_and_comments` | `pleading_type, applicant_names, sys_created_on` | 110,389 |
| Recent Public Notices | `x_fmc_ibfs_public_notices` | (not yet pulled — table name confirmed, fields not yet) | — |
| Recent Actions | (not yet found — fast-follow to locate) | — | — |

**Recent Pleadings & Comments is the confirmed wedge** — verified empirically, not just from the customer's description: pulled live data and `applicant_names` is **blank (`""`) on every row returned**. The field exists in the schema (aliased to display as "Filer" in the official UI) but isn't populated. The official portal genuinely cannot show who a pleading concerns without opening each document — exactly Stas's "impossible to navigate" complaint, now confirmed at the data level.

**Sequencing implication**: build **Recent Filings first**, not Pleadings & Comments. `applicant_name` is already structured and populated there, so the watchlist-matching mechanism (extract → resolve against `watchlist_entries`) can be proven end-to-end with zero document parsing. Pleadings & Comments comes second — it's the harder, higher-value case, since identifying the subject company requires actually opening the linked document and running it through the LLM extraction pipeline (the index gives us nothing to go on). That's also where the real product advantage over the official portal is sharpest, so it's worth building, just not first.

**Data quality gotcha, confirmed in both tables**: many `submission_date`/`sys_created_on` values are sentinel garbage, not real dates — e.g. `8888-08-08`, `4444-04-04`, `8800-08-22`, `3199-01-02`. The ingestion adapter must treat implausible years (e.g. before 1990 or after current year + 1) as `null` rather than parsing them literally into `news_articles.published_at`.

## Data model changes

New table: `watchlist_entries`
- `id` (PK)
- `label` (text, not null) — display name, e.g. "Viasat"
- `legal_name_normalized` (text, not null) — for matching against `extracted_entities`/`canonical_entities`
- `status` (text, not null, default `'watching'`) — `watching` | `pending`
- `created_at` (timestamptz, default now())

Schema convention / workflow: write the migration (`migrations/0015_add_watchlist_entries.sql`) and the corresponding `WatchlistEntry` model in `strata_core/models.py` as code. **Do not hand-generate `strata_core/schema.local.sql`** — per the existing convention (`docs/decisions.md`, 2026-02-07), that file is a literal `pg_dump` snapshot. The sequence is: (1) write the migration + model code, (2) run the migration against the dev DB, (3) regenerate the snapshot with `pg_dump strata --schema-only --no-owner > strata_core/schema.local.sql`. Steps 2–3 are run directly against the dev DB, not authored by hand.

`NewsSource` enum: add `FCC_ICFS` (and, if needed once the fetch mechanism is confirmed, a distinct value if ICFS turns out to need more than one logical source, e.g. a mirror vs. the official portal).

## Build order (from `to_build.md`)

1. Empty deployed skeleton — already done (app is live on Railway); this step is about confirming the new ingestion script + table deploy cleanly, not building UI yet.
2. Ingest everything, watch one name — `ingest_icfs.py` pulls all ICFS filings (unfiltered) into `news_articles` via the new adapter, through the existing extraction pipeline. `watchlist_entries` is seeded with just one row (Viasat) to prove the resolution/matching step against the full, real mirror.
3. Build the list + filter UI — a paginated page listing ingested filings, starting with **Recent Filings** (applicant_name already populated, fastest path to a working watchlist match), then **Recent Pleadings & Comments** (the harder case — applicant_names is empty on every row, so the subject company can only be identified by parsing the linked document through the LLM extraction pipeline). Dropdown filters to the hardcoded watchlist (Viasat); each row cites back to the source filing. A generated evidence-backed summary per filing is a fast-follow, not a blocker for v0.
4. Point at the customer's actual watchlist — once the mechanics work for Viasat, expand `watchlist_entries` to whatever names the customer actually wants monitored. Ingestion itself doesn't change at this step — it was already pulling everything.

## Access mechanism (confirmed live, 2026-06-23)

ICFS (`https://fccprod.servicenowservices.com/ibfs?id=ibfs_home`) is a ServiceNow Service Portal app — an AngularJS single-page app. The page shell has no data in it; everything renders client-side via AJAX calls back to the same host. There is no published/documented API — this was reverse-engineered from real network traffic. It is **public and unauthenticated** (`X-Is-Logged-In: false` on every response), but every data request still needs a valid session + anti-CSRF token, obtained by first loading the page like a browser would.

### Step 1 — bootstrap a session

```
GET https://fccprod.servicenowservices.com/ibfs?id=ibfs_home
```
- No special headers required beyond a normal `User-Agent`.
- Response sets session cookies via `Set-Cookie`: `JSESSIONID`, `glide_user_route`, `glide_node_id_for_js`, `BIGipServerpool_fccprod` (a load-balancer affinity cookie). All must be persisted and replayed on the next request — use a client with a cookie jar (e.g. `httpx.Client(...)`), not a one-off stateless request.
- The response HTML body embeds a CSRF-style token as a JS literal: `g_ck = '<64-char hex string>'`. Extract it with a regex (e.g. `g_ck = '([^']+)'`). This token is required on every subsequent POST — without it, the API returns `{"result":{..., "invalid_token": true}}` with no data, still as HTTP 201.

### Step 2 — fetch table data

```
POST https://fccprod.servicenowservices.com/api/now/sp/widget/842c213fdb4a0410d02ffb0e0f961934?id=ibfs_home
Content-Type: application/json;charset=UTF-8
Accept: application/json
X-UserToken: <g_ck from step 1>
Cookie: <cookies from step 1>
```
Every ICFS category is served by the **same widget endpoint and the same widget sys_id** (`842c213fdb4a0410d02ffb0e0f961934` — confirmed by swapping just the payload's `table`/`fields` between categories and getting back correctly different data). The category is selected entirely by the JSON body:

```json
{
  "table": "x_fmc_ibfs_base_table",
  "fields": "number,call_sign,applicant_name,submission_date",
  "o": "submission_date",
  "d": "desc",
  "p": 1,
  "filter": ""
}
```
- `table` / `fields`: per category — see the table above. `fields` is a comma-separated string, not a JSON array.
- `o` / `d`: sort column and direction (`asc`/`desc`).
- `p`: 1-indexed page number.
- `filter`: a ServiceNow encoded query string. Empty string = no filter = the **entire table**, which is how the 140,540/110,389 row counts were confirmed. The live site's default home-page widgets hardcode `filter` to a rolling 7-day window (e.g. `submission_dateONLast 7 days@javascript:gs.beginningOfLast7Days()@javascript:gs.endOfLast7Days()`) — we don't need to replicate that ServiceNow-specific syntax since an empty filter already gives us everything.

### Response shape

HTTP status is **201 Created** (not 200) even for a read-only query — don't treat non-200 as an error. Body:
```json
{
  "result": {
    "data": {
      "row_count": 140540,
      "num_pages": 7027,
      "window_size": 20,
      "p": 1,
      "list": [
        {
          "number": {"display_value": "SAT-LOA-20260108-00016", "value": "SAT-LOA-20260108-00016", "type": "string", "label": "File Number"},
          "applicant_name": {"display_value": "SpaceX Services, Inc.", "value": "SpaceX Services, Inc.", "type": "string", "label": "Applicant Name"},
          "submission_date": {"display_value": "2026-06-19 12:44:29", "value": "2026-06-19 16:44:29", "type": "glide_date_time", "label": "Date Filed"},
          "call_sign": {"display_value": "E260466", "value": "4a597d8e93a5cb10dec6791d48373c7c", "type": "reference", "label": "Call Sign/Auth ID"},
          "sys_id": "9cd5ba59cf60c7109d5eb42b1f851c98",
          "targetTable": "x_fmc_ibfs_form_312"
        }
      ]
    }
  }
}
```
- Every field except `sys_id`/`targetTable` is an object with `display_value` (human-readable) and `value` (raw — for `reference`-type fields like `call_sign`, this is a `sys_id` pointing at another table, not the human string; use `display_value` for that one).
- `targetTable` tells you which detail table the row actually lives in (`x_fmc_ibfs_form_312`, `x_fmc_ibfs_scl`, `x_fmc_ibfs_itc` observed so far for Filings) — needed to build the citation link.
- Page size (`window_size`) is fixed at 20 and wasn't controllable via any payload field we sent — not yet confirmed whether a larger page size is possible at all.

### Citation URLs (for the per-row "source filing" link)

- Filings (`x_fmc_ibfs_base_table`): `https://fccprod.servicenowservices.com/icfs?id=ibfs_application_summary&number=<file_number>` (uses the human-readable file number, e.g. `SAT-LOA-20260108-00016`).
- Pleadings & Comments (`x_fmc_ibfs_pleadings_and_comments`): `https://fccprod.servicenowservices.com/icfs?id=ibfs_pc_summary&sys_id=<sys_id>` (uses the row's `sys_id`, not a human number — confirmed from the page's own click-handler JS: `url.replace("<sys_id>", params.record.sys_id)`).
- Filer identity: there's a separate `ibfs_frn_summary&number=<FRN>` page keyed by FCC Registration Number — a stable numeric filer ID, not yet pulled into any list response we've seen. Worth investigating as a stronger entity key than name-matching once we're past v0's "Filings first" slice.

### Remaining open questions

- **Recent Actions** table name not yet found (only Filings, Pleadings & Comments, Public Notices confirmed so far) — quick to resolve the same way (grep the home page's widget config JSON).
- **Recent Public Notices** field list not yet pulled (table name `x_fmc_ibfs_public_notices` confirmed, but not its `fields` list) — same five-minute check as above.
- Whether page size is adjustable (would cut request counts for a full backfill) is unconfirmed — worth one more test before committing to ~7,027 requests for Filings and ~5,520 for Pleadings & Comments.
- Politeness/rate limits: no documented limit found. Given the request volume needed for a full backfill, the adapter needs a deliberate delay between requests and should be resumable (track progress, don't re-walk from page 1 on every run) rather than hammering a federal server in a tight loop.

## Definition of done (v0)

- `watchlist_entries` exists and is seeded with at least one entry (Viasat).
- `ingest_icfs.py` pulls real ICFS filings for that entry into `news_articles` (tagged `FCC_ICFS`), and they flow through the existing `fetch_html` → `clean_text` → `llm_raw` → `extract_entities` → `link_entities` pipeline without code changes to those steps.
- A paginated list UI shows ingested filings (at least Pleadings & Comments) with a working dropdown filter to the hardcoded watchlist (Viasat), each row citing back to the source filing.
- The customer (Stas) has seen it for at least one real name and given feedback.
